Unified Modeling Language (UML)

UML adalah bahasa pemodelan standar untuk menjelaskan, memvisualisasikan, merancang, dan mendokumentasikan sistem. Dengan UML kita dapat mempresentasikan struktur, perilaku, dan arsitektur fisik sistem menggunakan berbagai diagram. (Ringkasan materi diambil dari modul yang kamu upload).

🎯 Use Case Diagram

Use case menggambarkan fungsi sistem dari perspektif pengguna (aktor). Use case merepresentasikan scenario utama di mana aktor berinteraksi dengan sistem — misalnya "Lihat Denah", "Bayar SPP", "Cek Jadwal". Komponen penting: aktor, use case, dan system boundary. Use case berguna untuk menyusun requirement dan komunikasi rancangan dengan klien. Deskripsi Gambar Use Case : Siswa dapat membuka web, login, masuk ke beranda, melihat fasilitas (luar/tiap lantai), modul UML, serta kontak. Admin dapat login ke dashboard, memantau siapa yang sedang login, dan melakukan logout pengguna bila diperlukan

1. Aktor Siswa Aktor utama yang dapat mengakses sistem untuk melihat informasi dan fasilitas. Admin Aktor yang bertanggung jawab mengelola sistem serta memantau aktivitas login. 2. Use Case untuk Siswa Membuka Web Siswa membuka halaman utama sistem. Sistem menampilkan halaman login. Login Sebagai Siswa Siswa memasukkan kredensial (username & password). Jika berhasil, siswa diarahkan ke halaman Beranda. Masuk ke Beranda Siswa dapat mengakses berbagai menu dan modul yang tersedia di sistem. Mengakses Modul UML Siswa dapat membuka dan mempelajari modul UML yang tersedia di sistem. Mengakses Fasilitas Siswa dapat melihat berbagai fasilitas yang ada, yang terbagi menjadi beberapa area: Fasilitas Luar Lantai 1 Lantai 2 Lantai 3 Pada setiap area, siswa dapat menggunakan fitur Melihat Area untuk menampilkan detail fasilitas di area tersebut. Mengakses Menu Kontak Siswa dapat membuka halaman kontak untuk melihat informasi kontak pihak sekolah atau admin. Log Out Siswa dapat keluar dari sistem kapan saja. 3. Use Case untuk Admin Login Sebagai Admin Admin membuka halaman login dan memasukkan kredensial admin. Masuk ke Dasbor Admin Setelah login berhasil, admin diarahkan ke halaman Dasbor Admin. Melihat Apakah Ada yang Login Admin dapat memantau aktivitas login pengguna (misalnya, siswa yang sedang aktif). Log Out Admin keluar dari sistem. 4. Hubungan Antar Use Case “Login Sebagai Siswa” dan “Login Sebagai Admin” keduanya menuju ke proses Halaman Login. Setelah login, masing-masing aktor diarahkan ke halaman berbeda: Siswa → Beranda Admin → Dasbor Admin Use case “Log Out” dapat dilakukan baik oleh Siswa maupun Admin, dan akan kembali ke Halaman Login. Dari Beranda, siswa dapat berpindah ke berbagai menu seperti: Modul UML Kontak Fasilitas (Luar, Lantai 1–3) 5. Ringkasan Alur Utama Pengguna membuka web → Halaman Login. Login sebagai Siswa atau Admin. Setelah login: Siswa menuju ke Beranda → dapat menjelajahi fasilitas, modul, dan kontak. Admin menuju ke Dasbor → dapat melihat aktivitas login. Keduanya dapat melakukan Log Out untuk keluar dari sistem.

⚙️ Activity Diagram

Activity diagram menggambarkan alur kerja (flow) dari suatu use case, mirip flowchart tetapi mendukung perilaku paralel. Simbol penting: initial node, action/state, decision (diamond), fork/join (paralel), dan final node. Activity diagram berguna saat menjelaskan detil proses bisnis.

🧑‍🎓 Activity Diagram Siswa Deskripsi Umum: Activity Diagram Siswa menggambarkan proses yang dilakukan oleh siswa saat berinteraksi dengan sistem. Proses dimulai dari login, eksplorasi fitur yang tersedia, hingga logout. Alur Aktivitas: Siswa membuka web untuk mengakses sistem. Sistem menampilkan halaman login siswa. Siswa mengisi username dan password. Sistem melakukan validasi data login: Jika data valid, siswa lanjut ke langkah selanjutnya dan diarahkan ke halaman utama. Jika data invalid, sistem menampilkan pesan kesalahan "Username atau password salah". Setelah login berhasil, siswa diarahkan ke Halaman Beranda. Dari halaman ini, siswa dapat mengakses berbagai fitur, di antaranya: Melihat Fasilitas Luar Melihat Area Lantai, yang terdiri dari: Lantai 1 Lantai 2 Lantai 3 Melihat Modul UML Melihat Kontak Setelah selesai menggunakan sistem, siswa dapat melakukan Log Out untuk keluar dari akun. Kesimpulan Aktivitas Siswa: Diagram ini menunjukkan proses login dan navigasi fitur yang dapat dilakukan oleh siswa. Sistem memberikan akses hanya kepada pengguna yang berhasil melakukan autentikasi, dan siswa dapat menjelajahi informasi sesuai kebutuhan.

👨‍💼 Activity Diagram Admin Deskripsi Umum: Activity Diagram Admin menggambarkan proses yang dilakukan oleh admin dalam mengelola sistem. Proses ini dimulai dari login ke dalam sistem hingga melakukan pengelolaan data siswa dan logout dari sistem. Alur Aktivitas: Admin membuka web untuk mengakses sistem. Sistem menampilkan halaman login admin. Admin mengisi username dan password pada form login. Sistem melakukan validasi data login: Jika data valid, sistem mengizinkan admin lanjut ke langkah selanjutnya. Jika data invalid, sistem menampilkan pesan kesalahan "Username atau password salah". Setelah login berhasil, admin diarahkan ke Dashboard Admin. Dari dashboard, admin dapat mengakses Daftar Siswa, yang terbagi menjadi dua kategori: Siswa Terdaftar Siswa Belum Terdaftar Admin dapat melihat Status Siswa, termasuk siapa saja yang sedang login ke sistem. Admin memiliki dua opsi akhir: Log Out Siswa, untuk mengeluarkan siswa dari sistem. Log Out Admin, untuk keluar dari akun admin. Kesimpulan Aktivitas Admin: Activity Diagram ini menunjukkan bahwa admin memiliki peran penting dalam mengelola data dan memantau aktivitas siswa dalam sistem. Proses login menjadi tahap awal autentikasi sebelum admin dapat mengakses seluruh fitur pengelolaan

📶 Sequence Diagram

Sequence diagram menggambarkan interaksi antar objek/participant menurut waktu (atas → bawah). Garis vertikal = lifeline; kotak activation menunjukkan waktu eksekusi; panah mewakili message. Diagram ini berguna untuk menjelaskan alur pesan dalam sebuah scenario (use case).

Deskripsi Gambar Diagram : Siswa mengisi form login, sistem memverifikasi data, lalu jika berhasil, siswa diarahkan ke beranda dan datanya disimpan di Local Storage. Admin melakukan login serupa, kemudian masuk ke dashboard untuk mengambil daftar siswa, memeriksa siapa yang sedang login, serta dapat menghapus data LoggedUser untuk melakukan logout.

🧑‍🎓 Sequence Diagram Login Siswa Deskripsi Umum Diagram ini menggambarkan alur proses login dan interaksi antara Siswa, Halaman Login, dan Halaman Beranda pada sistem web siswa. Tujuan utamanya adalah untuk memverifikasi data login siswa dan menampilkan halaman beranda jika autentikasi berhasil. Alur Proses Siswa membuka website dan mengisi form login dengan username serta password. Halaman Login Siswa menerima input tersebut dan mengirimkan data ke Halaman Beranda untuk diverifikasi. Halaman Beranda memverifikasi data login dengan data siswa yang terdaftar. Hasil verifikasi (berhasil/gagal) dikirim kembali ke Halaman Login Siswa. Jika berhasil, sistem: Menyimpan data LoggedUser ke Local Storage. Menampilkan Halaman Beranda Siswa. Siswa kemudian dapat memilih menu yang tersedia (fasilitas, area, modul UML, atau kontak). Saat Siswa melakukan logout, sistem: Menghapus data LoggedUser dari Local Storage. Mengarahkan kembali ke Halaman Login Siswa. Inti Proses Autentikasi pengguna (siswa). Penyimpanan sesi login di Local Storage. Navigasi antar halaman (Login ↔ Beranda).

👨‍💼 Sequence Diagram Login Admin Deskripsi Umum Diagram ini menggambarkan alur login admin dan aktivitas manajemen data siswa di dalam sistem. Admin memiliki hak untuk melihat siapa yang sedang login dan mengelola daftar siswa. Alur Proses Admin membuka web dan mengisi form login dengan username serta password. Halaman Login Siswa mengirim data ke Halaman Beranda untuk proses verifikasi. Halaman Beranda melakukan verifikasi data admin. Hasil verifikasi (berhasil/gagal) dikembalikan ke Halaman Login Siswa. Jika berhasil, admin diarahkan ke dashboard. Dari dashboard, sistem: Mengambil daftar siswa dari Halaman Beranda. Mengembalikan daftar siswa ke halaman admin dan mengecek siapa yang sedang login (LoggedUser). Admin dapat melihat siapa yang sedang login. Ketika admin memilih untuk logout, sistem: Menghapus data LoggedUser. Mengembalikan tampilan ke Halaman Login Siswa. Inti Proses Autentikasi pengguna (admin). Pengambilan dan pengecekan data siswa yang sedang login. Logout dan penghapusan sesi admin. 🧭 Kesimpulan Umum Kedua sequence diagram tersebut menunjukkan dua peran berbeda: Siswa: Login untuk mengakses halaman beranda dan fitur pembelajaran. Admin: Login untuk mengelola data siswa dan memantau aktivitas login.

🏛️ Class Diagram

Class diagram menampilkan struktur statis sistem: kelas, atribut, operasi, dan relasi (association, aggregation, inheritance, realization). Class memiliki tiga area pokok: nama, atribut, dan metoda. Class diagram berguna untuk mendesain struktur data dan API internal.

Contoh kelas pada SmartSchool: User, Ruangan, Lantai, TransaksiBank.

🧩 1. Class: User (Superclass) Deskripsi: Kelas User merupakan superclass (kelas induk) yang digunakan sebagai dasar untuk kelas Admin dan Siswa. Kelas ini menyimpan data umum pengguna sistem seperti username dan password. Atribut: username: String — Nama pengguna untuk login ke sistem. password: String — Kata sandi pengguna. Method: login(): Boolean — Digunakan untuk proses autentikasi pengguna. logout(): void — Digunakan untuk keluar dari sistem. 🧑‍🎓 2. Class: Siswa (extends User) Deskripsi: Kelas Siswa merupakan turunan dari User yang mewakili pengguna berperan sebagai siswa. Siswa dapat melihat halaman beranda dan menjelajahi fasilitas sekolah. Atribut: idSiswa: int — Nomor identitas unik siswa. status: String — Menunjukkan status siswa (misalnya aktif/nonaktif). Method: lihatBeranda(): void — Menampilkan halaman beranda untuk siswa. lihatFasilitas(): void — Menampilkan daftar fasilitas sekolah. lihatArea(lantai: int): void — Menampilkan area tertentu berdasarkan lantai. 🧑‍💼 3. Class: Admin (extends User) Deskripsi: Kelas Admin merupakan turunan dari User yang berfungsi sebagai pengelola sistem. Admin memiliki hak akses untuk melihat data siswa dan mengelola status login siswa. Atribut: idAdmin: int — Nomor identitas unik admin. Method: tampilkanDashboard(): void — Menampilkan dashboard admin. lihatStatusSiswa(): void — Menampilkan status siswa yang sedang login. logoutSiswa(username: String): void — Mengeluarkan siswa tertentu dari sistem. 🔐 4. Class: SistemLogin Deskripsi: Kelas SistemLogin bertanggung jawab atas proses autentikasi pengguna (siswa dan admin) serta pengelolaan sesi login. Atribut: dataSiswa: List — Daftar data seluruh siswa. dataAdmin: List — Daftar data seluruh admin. Method: verifikasiSiswa(username: String, password: String): Siswa — Memverifikasi akun siswa berdasarkan username dan password. verifikasiAdmin(username: String, password: String): Admin — Memverifikasi akun admin. simpanSession(user: User): void — Menyimpan sesi pengguna yang login. hapusSession(user: User): void — Menghapus sesi pengguna yang logout. 🖥️ 5. Class: DashboardAdmin Deskripsi: Kelas DashboardAdmin merupakan antarmuka yang digunakan oleh admin untuk mengelola data siswa yang terdaftar di sistem. Atribut: daftarSiswa: List — Menyimpan daftar siswa yang terdaftar. Method: tampilkanDaftar(): void — Menampilkan daftar siswa. tampilkanStatusLogin(): void — Menampilkan status login siswa. logoutSiswa(username: String): void — Mengeluarkan siswa tertentu dari sistem. 🏫 6. Class: HalamanBeranda Deskripsi: Kelas HalamanBeranda merupakan tampilan utama yang dapat diakses oleh siswa untuk melihat menu, fasilitas, area, dan kontak sekolah. Method: tampilkanMenu(): void — Menampilkan menu utama di beranda. bukaFasilitas(): void — Menampilkan informasi fasilitas sekolah. bukaArea(lantai: int): void — Menampilkan area sekolah berdasarkan lantai. bukaKontak(): void — Menampilkan informasi kontak sekolah. 🔗 Relasi Antar Kelas: User adalah superclass untuk Admin dan Siswa (hubungan inheritance / pewarisan). SistemLogin berinteraksi dengan Siswa dan Admin untuk proses autentikasi dan manajemen sesi. Admin berinteraksi dengan DashboardAdmin untuk menampilkan dan mengelola data siswa. Siswa berinteraksi dengan HalamanBeranda untuk mengakses informasi fasilitas sekolah.

💾 Deployment / Development Diagram

Deployment diagram menggambarkan arsitektur fisik: node (server, client), artefak, dan bagaimana komponen ditempatkan di lingkungan runtime. Diagram ini penting untuk perencanaan infrastruktur dan instalasi sistem.

Contoh: web server (hosting HTML/CSS), database server, dan client browser yang mengakses halaman denah.

Langkah-langkah Penggunaan UML (Ringkasan)

1. Buat daftar business process level tertinggi.
2. Petakan use case untuk tiap business process dan lengkapi requirement.
3. Buat deployment diagram kasar untuk arsitektur fisik.
4. Definisikan requirement non-fungsional (security, performance).
5. Dari use case → buat activity diagram.
6. Identifikasi objek domain → buat sequence / collaboration diagram.
7. Rancang UI model untuk skenario use case.
8. Buat class diagram dan unit test untuk tiap class.
9. Kelompokkan class menjadi komponen → buat component diagram.
10. Perhalus deployment diagram (mapping komponen → node).
11. Bangun sistem (use case-driven atau component-driven).
12. Lakukan unit & integrasi test; sinkronkan model dan code.
13. Rilis perangkat lunak.
